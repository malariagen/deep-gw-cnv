Clarification — what is the band filter?

CRR (Copy Read Ratio) is the ratio of read depth inside a gene to the depth in the
surrounding 100 kb flanking region. Amplified genes should have CRR > 1.0. The problem
is that systematic biases (GC content, mappability, sample noise) can inflate CRR to
1.20–1.35 without any real copy gain. These false positives have no sustained copy
elevation in the genomic neighbourhood — the HMM does not assign them to a CN≥2 segment.

So calls in the range CRR [1.20, 1.35) are held for an extra check: what fraction of
the gene's bins fall inside a CN≥2 HMM segment? If the fraction is below a threshold
the call is demoted to CN=1 (not a real amplification). The "band" is this intermediate
CRR zone where the signal is ambiguous and needs HMM confirmation.

The "two-tier" part: exp 27 found that a single 50% threshold is too strict in the
upper sub-range [1.25, 1.35). Real GCH1 amplifications in that range have partial HMM
segment coverage — enough to confirm a real copy gain exists, but not 50% of gene bins.
So we split the band into two zones with different thresholds:

  Core zone   [1.20, 1.25):  ≥ 50% of gene bins in CN≥2 segments (artefacts here
                              are easily separated from TPs — the 0.50 threshold is clean)
  Extended zone [1.25, 1.35): ≥ 20% of gene bins in CN≥2 segments (lenient, to
                              recover weak-signal TPs while still blocking pure artefacts)
  Above 1.35:                unconditional pass (strong signal; no HMM check needed)

No data augmentation: confirmed. The codebase has an AugmentedNormalDataset class
that does Poisson resampling of normal samples, but it is only activated by
aug_normal_poisson: true in the config. Exp 29's config does not set that key, so
training runs on the raw read count data with log2 normalisation only.

Proposed experiment 29: 06_conv_vae trained on 400 bp core genome (dynamic n_bins)
Changes from exp 27 (last completed):
- architecture: 05_conv_vae → 06_conv_vae (same topology; n_bins auto-derived from data shape)
- store_path: 1000 bp core → 400 bp core (51986 bins vs 20814 bins; ~2.5× resolution)
- batch_size: 128 → 64 (wider inputs increase per-batch memory)
- Full retraining from scratch (linear layer dims change with bin count)
- Two-tier band caller (v11) already included — same thresholds as exp 28 proposal

What I expect:
GCH1 FNR should drop from 0.11 toward 0.04–0.08. At 400 bp, GCH1 spans ~5 bins
instead of ~2, giving the HMM a longer consecutive elevated run to commit to.
Combined with the lenient 20% threshold in the extended zone, marginal-CRR TPs in
[1.25, 1.35) should be reliably recovered. MDR1 and CRT are already well-resolved
at 1000 bp; no regression expected. PM2_PM3 FNs have CRR≈0 (coverage failures, not
resolution failures) — no improvement expected there.

What we could do instead:
- Run exp 28 (two-tier band only, 1000 bp model) first to isolate the threshold
  effect before committing to a full retrain on 400 bp data — cleaner attribution
  but slower path to a better GCH1 FNR.
- Increase HMM self_transition above 0.90 to reduce partial-coverage GCH1 FNs
  at 1000 bp before adding resolution, avoiding the full retrain cost.

Run time estimate: ~6–10 hours (full training + HMM + evaluation).
